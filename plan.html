Phase 1 — Fix GnuPG on Darwin

FIXDarwin needs pinentry-mac; the existing module only covers bash/zsh/fish init

What's already there

modules/home/security/gpg/default.nix (option custom.security.gpg) handles Fish interactiveShellInit, sets GPG_TTY and SSH_AUTH_SOCK, and writes ~/.gnupg/gpg{,-agent}.conf.

Problem: it doesn't set pinentry-program in gpg-agent.conf. On Darwin, pinentry-curses works in terminal but pinentry-mac is needed for GUI prompts and YubiKey touch. Without it, agent silently fails or falls back to nothing.

1.1 Update the GPG home module

Edit modules/home/security/gpg/default.nix — add a Darwin branch to gpgAgentConf:

let
  pinentryBin = if pkgs.stdenv.isDarwin
    then "${pkgs.pinentry_mac}/Applications/pinentry-mac.app/Contents/MacOS/pinentry-mac"
    else "${pkgs.pinentry-curses}/bin/pinentry-curses";

  gpgAgentConf = ''
    enable-ssh-support
    default-cache-ttl 28800
    max-cache-ttl 28800
    pinentry-program ${pinentryBin}
  '';

1.2 Enable in home profile

custom.security.gpg.enable = true;

1.3 Verify

gpg --card-status           # if YubiKey present
echo "test" | gpg --clearsign   # triggers pinentry-mac dialog
After initial YubiKey enrolment, set tools.git.enableSigning = true in the home profile.

Phase 5 — DNS Split Brain for VPN

FIXDarwin has no systemd-resolved — use /etc/resolver/ files instead

Problem

The existing openconnect module has a configureResolvedSplitDns block that calls resolvectl — this is Linux/systemd-only and is already guarded behind pkgs.stdenv.isLinux. The Darwin branch only handles routes, not DNS. After VPN connect, internal domains (pyn.ru, hh.ru, hhdev.ru) don't resolve.

Solution: /etc/resolver/ files

macOS reads per-domain DNS config from /etc/resolver/<domain> files. The myvpn up script needs to create these files, and myvpn down needs to remove them.

5.1 Update the Darwin section in openconnect module

Edit modules/home/security/openconnect/default.nix — extend the Darwin route_delete_command:

else if pkgs.stdenv.isDarwin then
  ''
    sudo route delete -net 192.168.0.0/16
    sudo route add -net 10.8.0.1/32 192.168.89.1
    # Split DNS: route internal domains to VPN nameservers
    for domain in pyn.ru hh.ru hhdev.ru; do
      sudo mkdir -p /etc/resolver
      printf 'nameserver 94.124.205.83\nnameserver 94.124.204.83\n' \
        | sudo tee /etc/resolver/$domain > /dev/null
    done
    sudo dscacheutil -flushcache
    sudo killall -HUP mDNSResponder
  ''

5.2 Teardown on myvpn down

In the down case of the script, remove the resolver files:

sudo kill -2 $(pgrep openconnect)
# Remove split DNS entries
for domain in pyn.ru hh.ru hhdev.ru; do
  sudo rm -f /etc/resolver/$domain
done
sudo dscacheutil -flushcache
sudo killall -HUP mDNSResponder

5.3 sudoers additions needed

Add mkdir, tee, rm on /etc/resolver/ to the passwordless sudo list in modules/darwin/system/security/default.nix:

/bin/mkdir -p /etc/resolver
/usr/bin/tee /etc/resolver/*
/bin/rm -f /etc/resolver/*
/usr/bin/dscacheutil -flushcache
/usr/bin/killall -HUP mDNSResponder

5.4 Verify

myvpn up
scutil --dns | grep -A3 "pyn.ru"   # should show VPN nameserver
dig internal-host.hh.ru            # should resolve
myvpn down
dig internal-host.hh.ru            # should fail or use public DNS
